### 第一幕：平台全局（3分钟）

**页面：登录 → 首页 Dashboard**

"各位老师好，我是青云的朱祚德。今天带大家快速看一遍 KubeSphere 容器平台。

（登录，首页亮相）

这就是平台首页。几个数字值得关注——

（鼠标指过去）

47 个企业空间，2 个集群统一纳管，48 个用户，44 个扩展组件按需加载。

设计理念是「微内核 + 可插拔」。核心平台很轻量，需要什么功能通过扩展中心一键安装——DevOps、服务网格、日志、监控，不需要的功能不装，不占资源。

最关键的一点：KubeSphere 完全兼容 kubectl 和 Kubernetes 原生 API，没有私有 CLI 命令，不会把你们锁定在我们的平台上。"

**操作：**
1. 登录，首页亮相
2. 鼠标依次指向：47企业空间 → 2集群 → 48用户 → 44扩展
3. 一句话带过：微内核、可插拔

**切换：** "先看集群——这是底座。"

### 第二幕：多集群管理（3分钟）

**页面：集群管理 → host 集群 → 节点**

"点击集群管理。

（进入集群列表）

两个集群——host 是主集群，member-1 是开发环境。一个控制台统一管多个集群，不管是测试、生产、还是边缘。

（进入 host → 节点）

host 集群 9 个节点，全都是 8核 16G 的 Worker 节点。状态全是绿色的运行中。

（鼠标划过节点信息）

每个节点的 CPU、内存、磁盘、Pod 数量实时可查。如果哪个节点负载高，一眼就能看到。

（切到 member-1 集群 → 节点）

同样的管理界面，同样的操作体验——这就是多集群统一纳管。将来你们如果有多个数据中心或者混合云，全部在同一面板管理。"

**操作：**
1. 首页 → 集群管理
2. 展示 host 和 member-1 两个集群
3. 进入 host → 节点，展示 9 节点全绿色
4. 切回 member-1，展示同样界面

**切换：** "集群看完了。资源怎么分给不同团队？这是企业最关心的问题。"

### 第三幕：多租户（5分钟）★ 差异化

**页面：企业空间 → poc 空间 → 项目列表**

"这是企业空间——

（进入企业空间列表，指 poc 空间）

47 个企业空间，每个对应一个团队或一个租户。比如「poc」这个空间给 PoC 测试团队用。

（进入 poc 空间）

空间里面可以建项目。比如 dev、test、prod——按环境拆分。

（点击空间设置 → 配额）

关键功能：资源配额。我可以在空间这一层设一个硬上限——CPU 不能超过 20 核，内存不能超过 40G。这样不管下面建多少项目，总量不会失控。

（点击项目配额）

每个项目还能再设配额——dev 给 8 核，test 给 8 核，prod 给 4 核。层层管控。

（切到空间成员）

再看权限——空间管理员只能管这个空间，完全看不到其他空间。同一个用户，在 A 空间是管理员，在 B 空间可以是只读。

这套多租户模型是 KubeSphere 跟 Rancher、OpenShift 最大的差异点——**四层角色 + 三层配额**，从平台到集群到空间到项目，权限边界清晰，资源不会互相侵占。"

**操作：**
1. 企业空间列表 → 指 poc 空间
2. 进入空间 → 展示项目列表
3. 空间设置 → 配额 → 展示 CPU/内存硬上限
4. 项目配额 → 展示项目级配额
5. 空间成员 → 展示角色分配

**切换：** "资源和权限分好了。来看业务怎么在上面跑。"

### 第四幕：工作负载全生命周期（5分钟）★ 核心

**页面：abctest 项目 → Deployment 列表**

"进入 abctest 项目——

（进入工作负载 → Deployment）

这里有两个 Nginx 应用在跑：nginx124 和 nginx02，各 2 个副本。注意他们的镜像版本不一样——一个是 trixie-perl，一个是 stable-perl。

（点击 nginx124）

先看这个——2 个副本都在 Running。

现在演示一下滚动更新。我把 nginx124 的镜像从 trixie-perl 换成 stable-perl。

（编辑 → 修改镜像 → 保存）

注意看 Pod 列表——不是一刀切全部停掉，是一个一个替换。老的 Pod 还在跑着，新的 Pod 已经起来了。替换完一个再换下一个。整个过程中业务不中断。

（等更新完成，所有 Pod 显示 stable-perl）

更新完成了，2 个 Pod 全是 stable-perl 了。

如果新版本有问题怎么办——

（点「回退」→ 选上一个版本 → 确认）

一键回滚。秒级恢复。这是生产环境最重要的能力——上线不害怕，出问题秒退。

再演示一个排障利器——

（点击某个 Pod → 终端）

Web 终端。直接浏览器里进容器，不需要 ssh 到宿主机再 docker exec。ls、ps、top、看日志——所有操作在这里完成。

（切到实时日志）

实时日志流，支持暂停、搜索、下载。运维排障不用上服务器。"

**操作：**
1. abctest → Deployment → nginx124
2. 编辑 → 改镜像 trixie-perl → stable-perl
3. 观察滚动更新过程（逐一替换）
4. 等待完成 → 执行回滚
5. 进入 Pod 终端 → 执行几个命令
6. 展示实时日志

**切换：** "单应用跑通了。但真正复杂的业务是微服务——来看 Bookinfo。"

### 第五幕：微服务与路由（4分钟）

**页面：stone-test 或 poc 项目 → Bookinfo**

"这是 Bookinfo，Istio 官方的微服务示例应用。

（进入项目 → 工作负载）

四个微服务：productpage、details、reviews、ratings。已经在跑。

（切到应用路由 / Ingress）

我们给它配了一个 Ingress 路由——

（点击 bookinfo-ingress）

指向 productpage 服务，通过 nip.io 域名可以直接从浏览器访问。

（打开 Bookinfo 页面）

这就是 Bookinfo 的页面。每次刷新，reviews 部分会显示不同的样式——因为 reviews 有三个版本在跑。

（刷新几次展示差异）

微服务架构的优势：每个服务独立部署、独立扩容，互不干扰。KubeSphere 平台提供完整的服务管理——从部署到路由到监控，一站式。"

**操作：**
1. 进入 stone-test 或 poc 项目
2. 展示 Bookinfo 四个微服务
3. 点击 Ingress 路由
4. 浏览器打开 Bookinfo 页面
5. 刷新 2-3 次展示 reviews 版本差异

**切换：** "服务跑起来了，但运维团队最需要的是——能看见、能告警。"

### 第六幕：可观测性（4分钟）

**页面：集群监控 → 节点监控 → 应用监控**

"KubeSphere 内置了全栈可观测性。

（进入集群监控）

第一层——集群级。9 个节点的整体 CPU、内存、磁盘、网络水位，一目了然。

（下钻到节点）

第二层——下钻到单个节点。demo-kse-1 的实时负载、网络流量、磁盘 IO。

（下钻到 Pod）

第三层——再下钻到 Pod。某个容器的 CPU 和内存历史趋势。

六层下钻：集群 → 节点 → 项目 → 工作负载 → Pod → 容器。出问题一路点下去就能定位，不用切好几个工具。

（切到日志查询）

日志——按项目、按关键词、按时间范围搜索。支持全文检索。

（切到告警策略——如果有的话）

告警——比如设一条「节点 CPU 超过 80%」的规则，触发后通过企业微信、钉钉、邮件通知。运维不需要一直盯着屏幕。"

**操作：**
1. 集群监控 → 展示整体
2. 点击一个节点 → 节点监控
3. 点击一个 Pod → 容器级监控
4. 日志查询 → 搜索演示
5. 告警策略 → 展示规则配置（如果有）

**切换：** "好，我快速总结一下。"

## FAQ 速查

| 问题 | 应答 |
|------|------|
| "和 Rancher 比？" | Rancher 强在多集群纳管。我们的差异化：四层多租户模型、内置 DevOps+服务网格可插拔、全开源 Apache 2.0（Rancher 许可证在收紧）。" |
| "和 OpenShift 比？" | "OpenShift 是标杆。三点差异：一，我们轻量——核心 2核4G；二，不绑操作系统——支持 CentOS/Ubuntu/麒麟/UOS；三，成本优势。" |
| "稳定吗？" | "山东港口科技集团生产环境跑三年，年均工单仅 6 个。" |
| "扩容？" | "加节点 → 一条命令加入集群 → 自动识别，业务不中断。" |
| "兼容 Jenkins？" | "Jenkinsfile 直接导入，Harbor 无缝对接，不强制替换现有工具。" |
